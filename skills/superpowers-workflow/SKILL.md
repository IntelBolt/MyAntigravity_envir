# Superpowers Development Workflow

Этот навык внедряет строгую методологию разработки, чтобы гарантировать качество кода, его тестируемость и соответствие требованиям. 

## 1. Socratic Brainstorming (Анализ требований)
Перед написанием любого кода, агент ДОЛЖЕН задать уточняющие вопросы, чтобы составить полную спецификацию (Spec).
- **Goal**: Понять «зачем» и «что», прежде чем решать «как».
- **Rules**:
  - Исследовать альтернативы.
  - Представить дизайн системы по частям для валидации пользователем.
  - Не переходить к планированию до утверждения спецификации.

## 2. Implementation Planning (Планирование)
После утверждения Spec, составить детальный план.
- **Goal**: Разбить работу на атомарные задачи.
- **Rules**:
  - Каждая задача должна занимать 2-5 минут.
  - Указывать точные пути к файлам.
  - Описывать шаги верификации для каждой задачи.
  - Следовать принципам YAGNI, DRY и KISS.

## 3. Test-Driven Development (TDD)
Цикл **RED-GREEN-REFACTOR** является обязательным.
- **RED**: Написать падающий тест. Убедиться, что он упал.
- **GREEN**: Написать минимально необходимый код, чтобы тест прошел.
- **REFACTOR**: Провести рефакторинг, сохраняя тесты зелеными.
- **Anti-pattern**: Запрещено писать функциональный код без предварительно написанного теста. Удаляй код, написанный до тестов.

## 4. Systematic Debugging (Системная отладка)
Если обнаружен баг, следовать 4-фазному процессу:
1.  **Reproduction**: Создать минимальный тест-кейс, воспроизводящий проблему.
2.  **Analysis**: Найти первопричину (Root Cause), используя логирование и трассировку.
3.  **Fixing**: Исправить проблему минимальным изменением.
4.  **Verification**: Убедиться, что тест-кейс проходит и нет регрессий.

## 5. Verification Before Completion
Перед завершением ветки разработки или задачи:
- Прогнать ВСЕ тесты проекта.
- Проверить код на соответствие Spec и Implementation Plan.
- Сделать самопроверку (Code Review) по чек-листу: типизация (TS), отсутствие лишних зависимостей, чистота кода.
